# SPEC — DEMO CONTENT PACK v1.0 (AXIOM_DEMO_UI)

## 0) Контекст и цель

На сайте уже существует рабочая архитектура и маршрутизация. В проект добавлены директории с исходниками контента:

* `temp/02.04_ECHELON`
* `temp/03_CHARACTERS`
* `temp/05_TECHNOLOGIES`
* `temp/06_FACTIONS`

Нужно:

1. **Удалить старые контент-заглушки** на сайте (stub/placeholder контент).
2. **Собрать минимальную стабильную демо-базу**: не переносить 100% инфы, а сделать **короткие, красивые, читаемые HTML** с выбранным контентом.
3. **Фреймворки и системные файлы не брать.**
4. **PNG-файлы можно копировать**, переименовывать и подключать на превью.
5. **Дубликаты запрещены**: если страница/файл/картинка уже есть в системе сайта — **повторно не обрабатывать**.

Результат: сайт без заглушек, с аккуратными демо-страницами (витрина контента).

---

## 1) Роли и ответственность (для агента)

Агент делает работу end-to-end в рамках существующей архитектуры:

* сначала **разбирается, как в проекте устроен контент** (где лежит, как подключается, как регистрируется в навигации/роутах, какие компоненты используются),
* затем **аудитит и удаляет заглушки**,
* потом **готовит демо-страницы** и интегрирует их по правилам ниже,
* в конце — **QA + отчёт**.

---

## 2) Входные данные

### 2.1 Источники контента (только эти каталоги)

* `temp/02.04_ECHELON/**`
* `temp/03_CHARACTERS/**`
* `temp/05_TECHNOLOGIES/**`
* `temp/06_FACTIONS/**`

### 2.2 Ассеты

* `*.png` внутри вышеуказанных деревьев.

---

## 3) Выходные данные (deliverables)

1. **Удалены** старые заглушки контента на сайте (и ссылки на них, если они висят в навигации/страницах).
2. **Сгенерирован и интегрирован DEMO CONTENT PACK**:

   * короткие HTML-страницы (или страницы в формате проекта, но контент должен быть **HTML-рендером**),
   * с превью-изображениями (PNG) там, где уместно.
3. **Манифест контента** (обязательно):

   * файл-таблица/JSON/MD, где указано:

     * источник (путь в `temp/...`)
     * назначение (куда положено на сайте)
     * slug/route/id
     * какие PNG использованы
     * отметка “уже существовало → пропущено”
4. Проект собирается без ошибок (build/lint, если есть).

---

## 4) Жёсткие правила и ограничения

### 4.1 Архитектуру НЕ менять/НЕ придумывать

* Не создаём “новую систему контента”.
* Работаем строго по текущим паттернам проекта: как уже сделаны существующие страницы/контент.

### 4.2 Не переносить 100% базы

* Делаем **витрину**: коротко, красиво, смыслово.
* Цель — “демка” и “вау-эффект”, а не архив.

### 4.3 Не брать фреймворки/системные файлы

Любые файлы, которые являются:

* системными регламентами/шаблонами,
* фреймворками,
* служебными README/спеками/навигацией/логами,
  — **не используем как источник демо-страницы**.

**Правило фильтра (эвристика):** пропускать файлы/папки, если в имени/пути есть маркеры типа:

* `FRAMEWORK`, `README`, `CHANGELOG`, `NAV`, `INDEX`, `TEMPLATE`, `SPEC`, `GMS`, `SYSTEM`, `STRUCT`, `.f_`, `00_` (служебное), и т.п.
  *(точный список агент фиксирует после изучения реальных имён в репо и добавляет в манифест как `ignore_patterns`)*

### 4.4 Никаких дублей

* Если в системе сайта уже есть страница/контент по этому объекту — **не делаем повторно**.
* Если PNG уже есть (или совпадает по смыслу) — **не копируем второй раз**.

**Обязательная логика дедупликации:**

* Сравнение по `slug`/`route`/`id` (как принято в проекте).
* Сравнение по имени файла (нормализованному).
* Для PNG: сравнение по имени + (если есть возможность) по размеру/хэшу (опционально, но желательно).

---

## 5) Требования к демо-объёму (минимальная стабильная база)

Цель — **минимум**, но чтобы выглядело “как продукт”.

**Таргет:** 8–16 страниц суммарно (в зависимости от реального количества годного материала).
Приоритет: качество > количество.

Распределение (ориентир):

* `02.04_ECHELON`: 2–4 страницы
* `03_CHARACTERS`: 3–5 страниц
* `05_TECHNOLOGIES`: 2–4 страницы
* `06_FACTIONS`: 1–3 страницы

Если материалов меньше — делаем меньше, но всё равно закрываем хотя бы **по 1 странице на раздел**.

---

## 6) Правила отбора “самая важная и интересная инфа”

Агент должен выбирать документы по приоритету:

**P0 (берём в первую очередь):**

* “Overview / концепт / описание сущности/локации/фракции/технологии”
* файлы с цельной подачей: что это такое + почему важно + особенности/факты
* файлы, которые дают сильный визуальный/сюжетный эффект (описания уровней, устройства, идеологии)

**P1 (если не хватает):**

* хорошо структурированные куски: списки, характеристики, краткие блоки

**P2 (не брать в демку, если только совсем пусто):**

* сырые черновики без смысла
* технические мета-документы
* системные шаблоны/правила/индексы

---

## 7) Формат демо-страниц

### 7.1 Страница должна быть “красивый HTML”

Не “простыня текста”. Обязательная структура:

1. **Hero**: название + короткий подзаголовок (1 строка)
2. **Teaser**: 4–8 строк “о чём это”
3. **Key Facts / Cards**: 3–6 карточек (коротко)
4. **Signature block**: атмосферный “excerpt” (цитата/фрагмент/описание)
5. **Gallery/Preview**: 1–3 PNG (если есть)
6. **Links/Related**: 3–8 связанных ссылок (если в проекте есть такой UI-паттерн; иначе — секция “Related” без ссылок)

### 7.2 Стиль

* Не узкие абзацы (нормальная ширина чтения).
* Минимум “воды”.
* Текст адаптировать под публичную демо-подачу: без внутренних служебных пометок и “мусорных” блоков.

### 7.3 Санитизация исходников

Если исходник Markdown/текст содержит:

* служебные метки,
* длинные секции навигации,
* повторяющиеся заголовки,
  — агент **должен вырезать/свернуть** это до демо-формата.

---

## 8) PNG: правила копирования и интеграции

1. PNG можно копировать из `temp/**`.
2. После копирования:

   * **переименовать** в единый стиль проекта (slug-based).
   * положить в **уже существующий** каталог ассетов (как принято в репо).
3. Запрещено:

   * дублировать PNG, если уже есть (по имени/хэшу/очевидному совпадению).
4. Каждая страница должна иметь:

   * `alt` текст
   * корректные пути
   * адекватные размеры/оптимизацию (если в проекте есть пайплайн — использовать его)

---

## 9) Удаление старых заглушек (обязательный этап)

### 9.1 Что считается заглушкой

* `Lorem ipsum`
* “Coming soon”
* “Stub / Placeholder”
* пустые страницы “раздел/контент скоро”
* временные блоки, которые не несут ценности

### 9.2 Что сделать

1. Найти заглушки поиском по проекту (ключевые слова + визуальный аудит страниц).
2. Удалить:

   * сами файлы (контент/страницы),
   * ссылки в навигации/роутах,
   * импорты, которые больше не используются.
3. Проверить, что после удаления:

   * нет 404/битых ссылок внутри демо-навигации,
   * билд проходит.

---

## 10) Пошаговый план работ (Agent Steps)

### Step A — Изучение архитектуры контента (без изменений)

* Найти:

  * где лежит контент (какие расширения, где html/страницы/контент-хаб),
  * как формируются routes/slugs,
  * как подключаются изображения,
  * есть ли единые компоненты для “article/page”.

**Артефакт:** короткая заметка “Content Flow Map” (3–10 пунктов).

### Step B — Аудит и удаление заглушек

* Список найденных заглушек → удалить → проверить ссылки.

**Артефакт:** `stub_cleanup_report` (список удалённого).

### Step C — Скан `temp/**` и отбор кандидатов

* Собрать список файлов-кандидатов.
* Применить ignore-patterns.
* Отобрать P0/P1.

**Артефакт:** `content_manifest` (черновик).

### Step D — Генерация демо-HTML и интеграция

* На каждый выбранный объект создать демо-страницу.
* Подключить PNG (если есть).
* Зарегистрировать страницу в существующей навигации/контент-реестре (как принято).

**Артефакт:** финальный `content_manifest`.

### Step E — QA

* Проверить:

  * страницы открываются,
  * вёрстка не ломается,
  * нет дублей ассетов,
  * нет заглушек,
  * build/lint ok.

---

## 11) Acceptance Criteria (критерии приёмки)

Задача считается выполненной, если:

1. **Все старые заглушки удалены** и больше не отображаются.
2. Есть **минимум 1 демо-страница** на каждый из 4 разделов:

   * Echelon / Characters / Technologies / Factions
3. Демо-контент выглядит как витрина:

   * hero + cards + excerpt + preview (если возможно)
4. **Дубликатов нет**:

   * не создано повторных страниц на уже существующие объекты
   * PNG не продублированы
5. Есть **content_manifest** с маппингом “источник → страница → ассеты”.
6. Проект **собирается** без ошибок.

---

## 12) Формат манифеста (рекомендовано)

Можно MD или JSON. Минимальные поля:

* `source_path`
* `category` (Echelon/Characters/Technologies/Factions)
* `title`
* `slug_or_route`
* `target_path_in_site`
* `assets_used[]`
* `status`: `created | skipped_existing | ignored_system | needs_review`
* `notes`

---

## 13) Важные примечания

* Если агент находит “серую зону” (файл вроде контентный, но похож на системный) — помечает `needs_review`, **не тащит его автоматически**.
* Если часть контента слишком большая — **делать компактную редактуру**, не копипастить простыню.
